DEVELOPER_ERROR (code 10)
sur Android — Analyse & Solution

@react-native-google-signin v16+ · play-services-auth 21.x · Expo SDK 52
Librairie : @react-native-google-signin/google-signin v16.1.2 Dépendance Android : play-services-auth:21.4.0 Résolu le : juin 2026

🔴 Symptôme

Sur un build Android production, la popup de sélection de compte Google s'affiche normalement. Dès qu'un compte est sélectionné, le SDK retourne immédiatement une erreur sans compléter la connexion :

DEVELOPER_ERROR — code 10
com.google.android.gms.common.api.ApiException: 10

Ce bug n'apparaît que sur Android. iOS (via expo-auth-session) fonctionne normalement.

🔍 Diagnostic — Ce qu'on a vérifié

DEVELOPER_ERROR (code 10) pointe habituellement vers une mauvaise configuration. Voici tout ce qui a été vérifié et qui était correct :

Point de vérification Statut
SHA-1 du keystore EAS enregistré dans Firebase ✓ Correct
Package name identique dans app.json et Google Cloud Console ✓ Correct
Client Android (type 1) et Web (type 3) dans le même projet Firebase ✓ Correct
google-services.json valide avec les deux clients ✓ Correct
apply plugin: 'com.google.gms.google-services' en dernière ligne du build.gradle ✓ Correct
Propagation Google attendue (5h+) ✓ Attendu
webClientId dans GoogleSignin.configure() ✗ Coupable

⚡ Cause racine

Depuis play-services-auth 21.x, le SDK Google valide plus strictement le webClientId passé dans GoogleSignin.configure(). Le client web auto-créé par Firebase Authentication (type 3) est rejeté avec DEVELOPER_ERROR, même s'il appartient au même projet que le client Android. L'erreur ne donne aucun détail — elle est silencieuse.

Subtilité : Le client web Firebase (généré automatiquement quand on active Google Sign-In dans la console Firebase) a des restrictions internes qui le rendent incompatible en tant que webClientId dans le SDK natif Android avec play-services-auth 21.x.

✅ Solution

Supprimer le webClientId de GoogleSignin.configure(). Sans ce paramètre, le SDK effectue le Sign-In en mode standalone et retourne toujours un accessToken utilisable par le backend.

❌ Avant — provoque DEVELOPER_ERROR

// play-services-auth 21.x rejette ça
GoogleSignin.configure({
  webClientId: '732343...apps.googleusercontent.com',
  scopes: ['openid', 'email', 'profile'],
});

✅ Après — fonctionne

// Sans webClientId = mode standalone
GoogleSignin.configure({
  scopes: ['email', 'profile'],
});

// accessToken toujours disponible
Impact sur le backend : Sans webClientId, le SDK ne retourne pas d'idToken mais l'accessToken reste disponible via getTokens(). Si ton backend utilise le flux OAuth2 standard avec access_token, aucun changement n'est nécessaire côté serveur.

📋 Configuration complète fonctionnelle

app.json

{
  "plugins": ["@react-native-google-signin/google-signin"],
  "android": {
    "googleServicesFile": "./google-services.json"
  }
}

google-services.json — les deux clients du même projet Firebase :

{
  "oauth_client": [
    {
      "client_id": "PROJECT_NUMBER-xxxx.apps.googleusercontent.com",
      "client_type": 1,         // Android
      "android_info": {
        "package_name": "fr.myjanaza.qabr",
        "certificate_hash": "SHA-1_DU_KEYSTORE_EAS"
      }
    },
    {
      "client_id": "PROJECT_NUMBER-yyyy.apps.googleusercontent.com",
      "client_type": 3          // Web (même projet)
    }
  ]
}

LoginScreen.js — configuration Android :

// Lazy require pour compatibilité Expo Go
let GoogleSignin = null;
if (Platform.OS === 'android') {
  try {
    ({ GoogleSignin } = require('@react-native-google-signin/google-signin'));
  } catch {}
}

// Dans useEffect — PAS de webClientId
if (Platform.OS === 'android' && GoogleSignin) {
  GoogleSignin.configure({
    scopes: ['email', 'profile'],
  });
}

// Sign-In
await GoogleSignin.hasPlayServices();
await GoogleSignin.signIn();
const tokens = await GoogleSignin.getTokens();
// tokens.accessToken est disponible — tokens.idToken sera null

🚀 Tester sans passer par le Play Store

Pour éviter les validations Play Store (1h+), utiliser la distribution interne EAS qui génère un APK installable directement via QR code, signé avec le même keystore que la production :

eas.json — ajouter le profil preview :

{
  "preview": {
    "distribution": "internal",
    "android": {
      "buildType": "apk"   // APK direct, pas AAB
    }
  }
}
eas build --profile preview --platform android
EAS fournit un QR code à scanner pour installer l'APK directement. Le SHA-1 du keystore est identique au build production — Google Sign-In fonctionne dans les mêmes conditions.

Résumé en une phrase

Avec @react-native-google-signin v16+ et play-services-auth 21.x, ne jamais passer webClientId dans GoogleSignin.configure() — ça déclenche silencieusement DEVELOPER_ERROR (code 10) à cause d'une validation stricte du client web Firebase.

L'accessToken reste disponible sans webClientId et suffit pour les backends OAuth2 standard.
